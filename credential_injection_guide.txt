CREDENTIAL INJECTION GUIDE FOR BROWSER AUTOMATION EVALUATION

Purpose: How to set up accounts and inject credentials so that

         actually testable with Browser-Use (and other frameworks).

WHY YOU NEED THIS
================================================================

Without credentials, your benchmark looks like this:

  Category            Tasks   Result Without Login
  ─────────────────────────────────────────────────────────────
  READ                82      ✅ Works — just navigate & extract
  CREATE              12      ❌ Always fails — needs login
  UPDATE               8      ❌ Always fails — needs login
  DELETE               6      ❌ Always fails — needs login
  FILE_MANIPULATION    5      ⚠️  Partly fails — download needs account

So 31 out of 113 tasks will auto-fail regardless of which
framework you test (Browser-Use, Skyvern, Steel), giving you
ZERO signal to differentiate between them.

The fix: inject credentials PER TASK so each framework can
actually attempt the action.


================================================================
PART 1: HOW BROWSER-USE HANDLES SENSITIVE DATA
================================================================

Browser-Use has a built-in `sensitive_data` parameter on the
Agent class. It works like this:

  from browser_use import Agent

  agent = Agent(
      task="Go to Amazon and add the first result to my cart",
      llm=llm,
      sensitive_data={
          "amazon_email":    "myaccount@gmail.com",
          "amazon_password": "MySecurePassword123"
      }
  )

HOW IT WORKS:
  - The values are NEVER included in the LLM's context window.
  - Instead, Browser-Use creates placeholder tokens like
    <amazon_email> and <amazon_password> that the LLM references.
  - When the agent actually needs to type the value into a browser
    field, the real value is injected at execution time.
  - This prevents your passwords from leaking into LLM logs.

EXAMPLE TASK PROMPT WITH SENSITIVE DATA:
  "Go to https://www.amazon.com. Log in using <amazon_email>
   and <amazon_password>. Then search for 'mechanical keyboard'
   and add the first result to the cart."

The LLM sees the placeholder; Playwright types the real value.

PART 2: HOW TO ADD CREDENTIALS TO YOUR EVALUATION SETUP
================================================================

STEP 1: Create a credentials file (DO NOT COMMIT TO GIT)
─────────────────────────────────────────────────────────
Create a file called `credentials.json` in your project root:

{
    "amazon": {
        "email":    "your-amazon-account@email.com",
        "password": "your-amazon-password"
    },
    "bestbuy": {
        "email":    "your-bestbuy-account@email.com",
        "password": "your-bestbuy-password"
    },
    "aliexpress": {
        "email":    "your-aliexpress-account@email.com",
        "password": "your-aliexpress-password"
    },
    "reddit": {
        "username": "your-reddit-username",
        "password": "your-reddit-password"
    },
    "github": {
        "username": "your-github-username",
        "password": "your-github-password"
    }
}

⚠️ ADD TO .GITIGNORE IMMEDIATELY:
   echo "credentials.json" >> .gitignore

STEP 2: Create a domain-to-credentials mapping
────────────────────────────────────────────────
You need a Python dict that maps a URL domain to the
correct credential keys. Example:

  DOMAIN_CREDENTIALS = {
      "amazon.com":     {"amazon_email": "...", "amazon_password": "..."},
      "bestbuy.com":    {"bestbuy_email": "...", "bestbuy_password": "..."},
      "aliexpress.com": {"ali_email":    "...", "ali_password":    "..."},
      "reddit.com":     {"reddit_username": "...", "reddit_password": "..."},
  }

STEP 3: Modify the eval_tasks.py Agent initialization
──────────────────────────────────────────────────────
Inside the _run_single() method (around line 842), change:

  BEFORE:
      agent = Agent(
          task=prompt,
          llm=llm,
          browser=browser,
          ...
      )

  AFTER:
      # Look up credentials for this domain
      from urllib.parse import urlparse
      domain = urlparse(target_url).netloc.replace("www.", "")
      creds = DOMAIN_CREDENTIALS.get(domain, {})

      agent = Agent(
          task=prompt,
          llm=llm,
          browser=browser,
          sensitive_data=creds if creds else None,
          ...
      )

STEP 4: Update task prompts to reference the placeholders
──────────────────────────────────────────────────────────
For tasks that require login, the prompt must tell the agent
WHICH placeholder to use. Modify how you build prompts so that
if a credential exists for this domain, the instruction is
prepended with a login step:

  def _build_prompt(url, instruction, creds):
      if creds:
          email_key = [k for k in creds if "email" in k or "username" in k]
          pass_key  = [k for k in creds if "password" in k]
          if email_key and pass_key:
              login_prefix = (
                  f"First, log in using <{email_key[0]}> "
                  f"and <{pass_key[0]}>. Then: "
              )
              return login_prefix + instruction
      return instruction

PART 3: PRACTICAL ACCOUNT SETUP STRATEGY
================================================================

Option A: Use Dedicated Throwaway Accounts (RECOMMENDED)
──────────────────────────────────────────────────────────
Create fresh accounts specifically for evaluation.
Pros:
  - Safe — your real accounts are untouched.
  - You can reset state between runs (empty cart, clear history).
  - You can share credentials with teammates.
Cons:
  - Takes ~30 minutes to set up accounts.
  - Some sites may require phone verification.

Sites commonly used in webbench_ready.csv to create accounts for:
  - Amazon (amazon.com)
  - AliExpress (aliexpress.com)
  - Best Buy (bestbuy.com)
  - Reddit (reddit.com)
  - GitHub (github.com)
  - Wikipedia (no account needed for many tasks)
  - Airbnb (airbnb.com)

Option B: Use Browser-Use Cloud Profiles (EASIEST)
────────────────────────────────────────────────────
Browser-Use Cloud has a "Profile Sync" feature. You:
  1. Log into your accounts in a local browser.
  2. Run: curl -fsSL https://browser-use.com/profile.sh | sh
  3. Get a cloud_profile_id.
  4. Pass that to the browser agent — it's already logged in!

Code:
  from browser_use import Browser, sandbox

  @sandbox(cloud_profile_id="your-profile-id")
  async def run_task(browser: Browser):
      agent = Agent(task=..., browser=browser, llm=llm)
      await agent.run()

This means you never need to inject passwords at all — the
cloud browser already has active sessions. This is the
most robust approach for a production-grade benchmark.

Option C: Pre-inject Session Cookies (ADVANCED)
─────────────────────────────────────────────────
Instead of passwords, inject browser session cookies. This
bypasses login pages entirely (faster, more stable).

  browser = Browser(
      storage_state="amazon_cookies.json"  # exported from your browser
  )

To export cookies from Chrome:
  Use the "EditThisCookie" extension to export as JSON.

PART 4: WHICH TASKS NEED WHICH CREDENTIALS
================================================================

Looking at webbench_ready.csv categories that need login:

CREATE tasks (need account to post/create):
  - Adding items to cart (Amazon, AliExpress, BestBuy)
  - Creating posts/threads (Reddit, CBR forums)
  - Submitting forms with user data

UPDATE tasks (need account to modify):
  - Changing shipping address (BestBuy, APA)
  - Editing forum posts
  - Updating profile settings

DELETE tasks (need account to remove):
  - Removing wishlist items (Barnes & Noble)
  - Deleting forum threads
  - Removing promo codes from cart

FILE_MANIPULATION tasks:
  - Downloading ebooks/files (may need premium account)
  - Exporting CSV/Excel reports (usually need premium on Crunchbase)

PART 5: THE FINAL ARCHITECTURE (RECOMMENDED)
================================================================

The cleanest setup for your evaluation is:

  1. Create dedicated throwaway accounts for ~5-6 sites.
  2. Store credentials in credentials.json (gitignored).
  3. Load them in eval_tasks.py and pass them as sensitive_data.
  4. Run all 5 categories with full credential support.

Your result CSV will then have:
  - Real SUCCESS/FAILED signals across ALL categories.
  - Meaningful comparison between Browser-Use, Skyvern, Steel.
  - Firewall/proxy blocks properly isolated as ENVIRONMENT_LIMIT.
  - Agent reasoning failures properly isolated as AGENT_FAILURE.

Without this, only READ tasks are meaningful for comparison.
