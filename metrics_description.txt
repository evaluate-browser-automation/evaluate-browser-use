# Evaluate Browser-Use: Key Metrics Description

This document explains the meaning of each column/metric that is output into the results CSV by `eval_tasks.py`. These metrics help you diagnose exactly why an agent failed or succeeded, especially when dealing with complex networking environments like corporate firewalls.

## 1. Identity & Task Setup
- `task_id`: Unique identifier for this specific task run (e.g., task_0001).
- `run_timestamp`: The exact date and time the task started.
- `category`: The type of web action being tested (READ, CREATE, UPDATE, DELETE, FILE_MANIPULATION).
- `url`: The starting URL given to the agent.
- `url_domain`: The root domain extracted from the URL (e.g., amazon.com).
- `instruction`: The exact prompt/goal provided to the agent.
- `instruction_length`: The character count of the instruction.

## 2. Execution & Performance
- `status`: The final outcome of the task (SUCCESS or FAILED).
- `latency_seconds`: Total wall-clock time the agent spent executing the task.
- `steps_taken`: How many actions (e.g., click, type, scroll) the agent successfully executed.
- `steps_budget`: The maximum number of actions the agent was allowed to take before being cut off.
- `steps_utilization_pct`: How much of the allowed budget was used (steps_taken / steps_budget).
- `retry_count`: The number of times the framework automatically retried a failed step.

## 3. LLM & Reasoning Analytics
- `llm_model`: The specific AI model used (e.g., gpt-4o, DeepSeek-R1-Distill, gemini-2.0-flash).
- `llm_provider`: The backend system providing the LLM (e.g., local or gemini).
- `llm_calls`: Total number of separate API requests made to the LLM during the task.
- `total_thought_chars`: The total character length of the agent's internal "thinking" or reasoning blocks. High numbers indicate the agent was struggling or looping.
- `final_answer`: The final text output, extraction, or conclusion provided by the agent.
- `final_answer_length`: The character count of the final answer.
- `avg_step_latency_seconds`: The average time it took for the LLM to process a single action step.
- `max_step_latency_seconds`: The absolute longest delay for a single step (useful for finding LLM timeout bottlenecks).

## 4. Error Diagnostics
- `error_code`: A standardized category for why the task failed (e.g., `Security_FirewallBlocked`, `Auth_CredentialInjectionFailed`, `Logic_AgentLoopOrHallucination`).
- `error_label`: A human-readable description of the error code.
- `error_hint`: A suggestion on how to fix or bypass the specific error.
- `error_message`: The raw, underlying error thrown by Playwright or the Browser-Use framework.

## 5. Obstacle Signals (Trace Analysis)
These metrics are `True` or `False`. They are generated by scanning the agent's raw trace logs for specific keywords.
- `has_captcha_signal`: True if the agent encountered "captcha" or "human verification" popups.
- `has_auth_signal`: True if the agent got stuck on "login", "sign in", or password screens.
- `has_timeout_signal`: True if the agent timed out or took 0 steps.
- `has_bot_block_signal`: True if the target website actively rejected the agent (e.g., HTTP 403, "access denied").
- `has_firewall_signal`: True if your corporate proxy/firewall intercepted the connection (e.g., Zscaler, Palo Alto, Proxy auth, or SSL certificate stripping errors).

## 6. Root Cause & Task Validity
These metrics help you filter out tasks that are "unfair" or impossible for the agent to solve.
- `failure_root`: The highest-level reason for failure. It can be:
   - `TASK_INVALID`: The URL is dead or the product no longer exists.
   - `FRAMEWORK_LIMIT`: The agent lacks a required feature (e.g., no password given, no captcha solver).
   - `ENVIRONMENT_LIMIT`: Blocked by a corporate firewall or proxy.
   - `AGENT_FAILURE`: The site was reachable and fair, but the agent hallucinated or got confused.
- `task_validity_code`: A more specific reason for invalidity (e.g., `URL_Dead`, `Framework_NoCredentials`).
- `task_validity_reason`: A detailed text explanation of why the task was marked invalid.
- `task_validity_confidence`: A score (0.0 to 1.0) indicating how confident the evaluator is in its validity assessment.

## 7. Pre-flight URL Probing
Before the browser even launches, the script "pings" the URL to see if it's alive.
- `url_http_status`: The HTTP status code returned by the initial ping (e.g., 200 = OK, 403 = Forbidden, 407 = Proxy Auth, 0 = Unreachable/SSL Error).
- `url_probe_note`: Extra context from the ping, such as "Redirected to X" or "SSL/Proxy Error".
